# システム要件定義書

## システム概要

**e-library Next** は、自動車メーカー向けの技術情報活用プラットフォームです。既存の情報閲覧サービス「e-library」を進化させ、**整備士が「探す」という行為をしなくても、作業フローの中で必要な情報が自動的に届く**仕組みを実現します。整備士の作業工程（入庫受付 → 診断 → 作業計画 → 作業実施 → 確認 → 納車）を起点に、各タイミングで必要なマニュアル、TIE（修理事例レポート）、注意事項、類似事例を自動提示します。検索機能は、自動提示でカバーできなかった場合のフォールバック（補完機能）として位置づけます。これにより、整備士の自己解決力向上、作業時間短縮、メーカーの販売店支援強化、現場知見の資産化を実現します。既存の10年以上の運用実績と約160ユーザーの利用データを基盤に、「情報を探すシステム」から「情報が届くシステム」へパラダイムシフトし、収益性を大幅に向上させることを目指します。

### 1. システムの目的

#### 解決する具体的な課題

**整備士・技術者の課題:**
- **「探す」という行為そのものが作業時間を圧迫している**（現状：情報探索に1作業あたり10-20分）
- 必要な情報がどのマニュアルにあるか分からない（探索負荷）
- キーワードが合わないと検索できない
- 症状から原因や手順を探しにくい（判断負荷）
- 過去の類似事例を探すのに時間がかかる（共有負荷）
- 新人や経験の浅い技術者ほど自己解決が難しい（教育負荷）
- **作業フローが中断され、集中力が途切れる**

**サービスショップ責任者の課題:**
- 技術者ごとの理解度や困りごとが見えない（管理負荷）
- 同じような質問・トラブルが繰り返される
- 教育すべきテーマを把握しにくい

**自動車メーカーの課題:**
- 販売店がどの情報で困っているか把握しづらい
- マニュアルが使われているか、理解されているか分からない
- 市場不具合や教育不足の兆候を早期に把握しにくい
- TIEや問題交流の知見が十分に再利用されていない
- 販売店支援・品質改善・教育施策にデータを活かしきれていない

#### 目標とする成果

- **「探す」時間を80%削減**: ワークフロー駆動の自動情報提示により、検索する必要性そのものを削減（目標：1作業あたり2-4分）
- **作業フローの中断をゼロに**: 作業の各工程で必要な情報が自動的に表示され、作業に集中できる
- **自己解決率を50%向上**: 作業タイミングに応じた情報提示により、問い合わせ前の自己解決を促進
- **販売店課題の可視化**: 分析ダッシュボードにより、メーカーが販売店の困りごとを定量的に把握
- **収益性の向上**: 月額約2,000円から、ユーザー数課金とEnterprise分析機能により、月額数十万円規模へ拡大

#### 期待されるビジネスインパクト

- **整備現場**: 作業効率向上、品質向上、新人教育の効率化
- **販売店経営**: 顧客対応品質向上、手戻り削減、技術者の自己解決力強化
- **自動車メーカー**: 販売店支援の高度化、品質改善の迅速化、教育施策の最適化、市場不具合の早期発見

### 2. 主な機能

#### コア機能（必須）：ワークフロー駆動の自動情報提示

1. **作業コンテキスト認識エンジン**
   - 整備士の現在の作業工程を自動認識（入庫 → 診断 → 作業計画 → 作業実施 → 確認 → 納車）
   - 車両情報（車種、年式、VIN）、DTC、症状、作業履歴を統合的に把握
   - 診断機・整備受付システムとの連携（将来：Phase 4-5で実装）
   
2. **工程別自動情報提示機能**
   
   **入庫受付時:**
   - 該当車種・年式のよくあるトラブルと注意事項を自動表示
   - 過去の同一車両の作業履歴（自ディーラー内）
   
   **診断時（DTC読み取り後）:**
   - DTCに対応する診断手順、原因候補、確認項目を自動表示
   - 類似DTCのTIE事例、問題交流の解決済み質問を自動表示
   - 見逃しやすい注意事項・警告を強調表示
   
   **作業計画時:**
   - 必要なマニュアルページ（作業手順、トルク値、部品番号）を自動表示
   - 必要な工具、部品、作業時間の目安
   - 関連する追加確認事項（「この作業時はここも確認すべき」）
   
   **作業実施時:**
   - 作業中の注意事項・警告を常時表示（安全確認、禁止事項）
   - ステップバイステップの作業手順（現在のステップをハイライト）
   - 類似作業でのトラブル事例（「この作業でよくあるミス」）
   
   **確認時:**
   - 作業完了後の確認項目チェックリスト
   - 動作確認手順、試運転時の確認ポイント
   
   **納車時:**
   - 顧客への説明事項（「今回の修理内容と今後の注意点」）

3. **プロアクティブレコメンデーション**
   - 過去の作業履歴に基づく情報提示（「前回のこの作業では、この情報を参照しました」）
   - 他の整備士の作業パターンに基づく提案（「同じ症状の作業では、この情報がよく使われています」）
   - 季節・時期に応じた情報提示（「この時期によくあるトラブルです」）

4. **TIE管理機能**
   - TIEレポート作成フォーム（車種、年式、症状、部位、発生状況、原因、対応プロセス、結論）
   - メーカー承認ワークフロー（Draft → Submitted → Under Review → Approved → Published）
   - 写真・動画の添付、関連マニュアルとのリンク

5. **問題交流機能**
   - 質問投稿時の類似質問・関連マニュアルの自動提示（重複質問削減）
   - 回答一覧とベストアンサー選定
   - メーカー確認依頼とTIE化候補の推薦
   - 解決済みステータスとタグ管理

#### 差別化機能（競合優位性）

1. **コンテキストアウェアAI**
   - 作業工程、車両情報、DTC、症状を統合的に理解し、最適な情報を自動提示
   - マニュアル・TIE・問題交流の横断要約（参照元リンク必須表示）
   - 確認すべき手順と注意点の自動抽出
   - 作業中の「今必要な情報」をリアルタイムに予測

2. **メーカー向け分析ダッシュボード**
   - MAU、閲覧数、検索数、投稿数の利用状況分析
   - 車種別課題、頻出症状、急増している症状の可視化
   - 未解決課題の一覧と優先度の提示
   - 教育テーマ抽出とマニュアル改善候補の提案
   - ディーラー別利用状況と支援が必要な店舗の特定
   - **自動提示の精度分析**（自動提示された情報がどの程度活用されたか）

3. **現場知見の資産化**
   - 問題交流からTIEへの格上げワークフロー
   - 類似事例のグルーピングとナレッジグラフ化
   - 解決記録による有用情報の蓄積
   - 作業パターンの学習と改善（使われる情報、使われない情報の分析）

#### フォールバック機能（補完機能）

**AI統合検索機能**（自動提示でカバーできなかった場合のみ使用）
   - マニュアル、TIE、問題交流を横断した全文検索
   - 自然言語検索（「エンジンがかからない時はどうする？」のような質問形式）
   - 車種、年式、部位、症状、DTC、キーワードでの絞り込み
   - 検索結果の関連度スコア表示と情報種別ラベル表示
   - ベクトル検索（pgvector + OpenAI Embeddings）による意味的類似検索

#### 付加価値機能（将来拡張）

1. **教育モジュール**
   - 頻出課題から学習コンテンツを自動生成
   - 習熟度管理と進捗トラッキング

2. **多言語対応**
   - 日本語、英語、中国語、タイ語などへの自動翻訳
   - 音声入力・音声読み上げ機能

### 4. 実現方法の概要

#### 主要なテクノロジー要素

- **フロントエンド**: Next.js (React) + TypeScript（モダンなUI/UX、サーバーサイドレンダリング、リアルタイム更新）
- **バックエンド (BaaS)**: Supabase（認証、データベース、リアルタイム機能、ストレージを一元管理）
- **データベース**: Neon (PostgreSQL)（スケーラブルなサーバーレスPostgreSQL、全文検索、JSON対応）
- **ホスティング**: Vercel（自動デプロイ、CDN、エッジ最適化）
- **AI機能**: OpenAI API / Claude API（コンテキスト理解、情報抽出、要約、類似事例抽出）
- **検索基盤**: PostgreSQL全文検索 + ベクトル検索（pgvector）（フォールバック時の意味的類似検索）
- **作業コンテキスト認識**: 作業工程、車両情報、DTC、症状、履歴を統合的に管理するコンテキストエンジン

#### 重要なインテグレーションポイント

- **既存e-libraryからのデータ移行**: マニュアル、TIE、問題交流の既存データの構造化とタグ付け
- **メーカー承認ワークフローとの統合**: TIE承認プロセスのシステム化
- **作業工程管理との統合**: 整備士の現在の作業工程を把握し、適切な情報を自動提示
  - Phase 1-2: 手動入力（整備士が作業工程を選択）
  - Phase 3-4: 半自動（DTCやVIN入力により作業工程を推定）
  - Phase 5: 全自動（診断機・整備受付システムとのAPI連携により作業工程を自動認識）
- **外部システム連携（Phase 4-5）**: 
  - 診断機連携（DTC自動取得、車両情報自動取得）
  - 整備受付システム連携（入庫情報、作業指示書）
  - DMS連携（ディーラーマネジメントシステム）
  - メーカー品質システム連携（品質データの双方向連携）

#### スケーラビリティへの考慮

- **初期想定**: 10ディーラー、160ユーザー → 3秒以内のレスポンスタイムを目指す
- **1年後目標**: 100ディーラー、1,600ユーザー → Supabase + Neon + VercelのスケーラビリティとCDNを活用
- **3年後目標**: 複数メーカー展開、10,000ユーザー規模 → データベースシャーディング、読み取りレプリカ、キャッシング戦略の導入

## 情報の前提と仮定

### 分析に使用した情報源

- `docs/input/e-library_service_overview.md`: 既存e-libraryの概要、利用状況、主な機能、利用ログ
- `docs/input/e-library_next_mockup_document.md`: e-library Nextへの進化計画、ビジネスモデル、7 Powers分析、プロダクト情報、制約条件、開発ロードマップ

### 情報が不足している場合に置いた仮定

#### 非機能要件に関する仮定

- **ターゲットユーザー規模**: 初期1,000人規模（現状160ユーザー + 新規ディーラー展開）を想定し、主要画面のレスポンスタイムは3秒以内、検索は1秒以内を目指す
- **可用性**: 99.5%以上（月間ダウンタイム3.6時間以内）。業務時間帯（平日9:00-18:00）は特に高い可用性を確保
- **同時アクセス**: ピーク時100同時接続を想定（1ユーザーあたり1日10分利用、業務時間帯集中と仮定）

#### 技術スタックに関する仮定

- 特に指定がなかったため、プロンプトの推奨に従い、Next.js + TypeScript + Supabase + Neon + Vercelを採用
- **選定理由**: 
  - 開発効率重視（BaaSによる認証・DB・ストレージの統合管理）
  - スケーラビリティ（サーバーレスアーキテクチャ）
  - コスト最適化（初期規模では低コスト、スケール時は従量課金）
  - 保守性（モダンなTypeScriptエコシステム、豊富なコミュニティ）

#### データ構造に関する仮定

- 既存マニュアル、TIE、問題交流のデータは一定程度構造化されているが、車種・年式・症状・部位・DTCのタグ付けは不完全と仮定
- Phase 1でデータ構造化とタグ付けを優先的に実施する必要がある

#### チームスキルに関する仮定

- UI/UX設計、フロントエンド開発、バックエンド開発の基本スキルは保有
- 自動車整備業務の理解は、既存e-libraryの運用実績から一定のドメイン知識があると仮定
- AI活用については、OpenAI/Claude APIの利用経験が必要だが、外部サポートや学習で対応可能

## 機能要件

### 0. 作業コンテキスト管理（最重要コア機能）

#### 0.1 作業セッション管理
- **作業開始**: 整備士が作業を開始する際に、車両情報（車種、年式、VIN）と作業内容を入力
  - Phase 1-2: 手動入力（ドロップダウン + 自動補完）
  - Phase 3-4: VIN入力による自動車両情報取得
  - Phase 5: 整備受付システムとの連携により自動取得
  
- **作業工程管理**: 現在の作業工程を管理
  - 入庫受付 → 診断 → 作業計画 → 作業実施 → 確認 → 納車
  - 整備士が工程を進める際に、手動で更新（Phase 1-3）
  - AIによる工程推定（Phase 4以降：閲覧情報や経過時間から工程を推定）
  
- **コンテキスト情報の統合管理**:
  - 車両情報（車種、年式、VIN、市場、仕様）
  - DTC（診断トラブルコード）
  - 症状（入庫時の顧客申告、整備士の診断結果）
  - 作業履歴（過去の同一車両の作業記録）
  - 閲覧履歴（今回の作業で参照したマニュアル、TIE、問題交流）

#### 0.2 コンテキストベース情報推薦エンジン
- **推薦ロジック**:
  1. 作業工程 × 車両情報 × DTC × 症状 に基づく情報抽出
  2. 類似作業パターンの分析（他の整備士の作業履歴から学習）
  3. 過去の自分の作業履歴に基づく個人最適化
  4. 季節・時期・地域に応じた情報の優先順位付け
  
- **推薦情報の種別**:
  - 公式マニュアル（該当ページを直接表示）
  - TIE事例（類似症状・DTC）
  - 問題交流の解決済み質問
  - 注意事項・警告（見逃しやすい重要情報）
  - 確認項目チェックリスト
  - 必要工具・部品リスト
  
- **推薦精度の学習**:
  - 整備士が参照した情報を記録
  - 推薦された情報が役立ったかのフィードバック収集
  - 機械学習による推薦精度の継続改善

#### 0.3 工程別自動情報提示

**入庫受付時の自動提示:**
- 該当車種・年式のよくあるトラブルTOP5
- 最近増加している症状（この車種でのトレンド）
- 過去の同一車両の作業履歴（自ディーラー内）
- リコール・サービスキャンペーン情報

**診断時（DTC読み取り後）の自動提示:**
- DTCの意味と考えられる原因
- DTC診断フローチャート（該当マニュアルページ）
- 類似DTCのTIE事例（症状、原因、解決策）
- 問題交流の類似質問と解決済み回答
- 確認すべき関連部位・センサー
- 見逃しやすい注意事項（「このDTCの場合、○○も必ず確認」）

**作業計画時の自動提示:**
- 必要なマニュアルページ（作業手順、トルク値、部品番号、配線図）
- 必要な工具リスト
- 必要な部品リスト（部品番号、在庫確認リンク）
- 標準作業時間の目安
- 関連する追加確認事項（「この作業時は○○も点検推奨」）
- よくある作業ミスの注意喚起

**作業実施時の自動提示:**
- ステップバイステップの作業手順（現在のステップをハイライト）
- 各ステップの注意事項・警告（常時表示、強調表示）
- トルク値、締付順序などの重要数値
- 類似作業でのトラブル事例（「この作業でよくあるミス」）
- 作業中の確認ポイント（「○○の状態を確認してください」）

**確認時の自動提示:**
- 作業完了後の確認項目チェックリスト
- 動作確認手順
- 試運転時の確認ポイント
- よくある確認漏れの注意喚起

**納車時の自動提示:**
- 顧客への説明事項（今回の修理内容、原因、対応）
- 今後の注意点（「○○に注意してください」）
- 次回点検時期・推奨メンテナンス

#### 0.4 プロアクティブ通知
- 作業工程が一定時間進まない場合の支援提案（「何か困っていることはありますか？」）
- 新しい関連TIE・マニュアル更新の通知（作業中の車種・症状に関連）

### 1. 認証・権限管理

  - ディーラー管理者（自ディーラーのユーザーを管理）

  - メーカー別データ分離（テナント分離）

### 3. AI統合検索機能（フォールバック機能）

**重要**: この機能は、作業コンテキストベースの自動情報提示でカバーできなかった場合の補完機能として位置づけます。理想的には、整備士は作業セッション内の自動提示情報だけで作業を完了できるため、検索機能を使う必要がありません。

#### 3.1 検索インターフェース
- **検索バー**: 自然言語検索（「エンジンがかからない時はどうする？」のような質問形式）
  - サジェスト機能（入力中に候補を表示）
  - 音声入力対応（Phase 5）
  
- **現在の作業コンテキストを引き継ぎ**:
  - 作業セッション中の検索は、現在の車種・年式・症状を自動的に絞り込み条件として設定
  - 「現在の作業に関連しない情報を探す」チェックボックス（絞り込みを解除）
  
- **絞り込み条件**（手動で調整可能）:
  - 車種（複数選択可）
  - 年式（範囲指定）
  - 症状（プルダウン + 自由入力）
  - 部位（プルダウン）
  - DTC（入力補完）
  - マニュアル種別（サービス、オーナーズ、新車トレーニング、車体修理）
  - 情報種別（マニュアル、TIE、問題交流、ニュース）

#### 3.2 検索結果表示
- **表示項目**:
  - タイトル
  - 情報種別ラベル（公式マニュアル、TIE承認済み、問題交流、ニュース）
  - 関連度スコア（★★★☆☆形式）
  - 車種、年式、症状、部位のタグ
  - 最終更新日
  - スニペット（検索語ハイライト）
  
- **ソート**: 関連度順、更新日順、閲覧数順
- **フィルタ**: 検索結果のさらなる絞り込み（メーカー承認済みのみ、解決済みのみ、など）

#### 3.3 関連情報の表示
- よく一緒に見られている情報（レコメンデーション）
- 類似の検索クエリ
- 関連する注意・警告事項

#### 3.4 検索フィードバック収集
- 「探している情報は見つかりましたか？」（Yes/No）
- No の場合: 「どのような情報を探していましたか？」（自由入力）
- このフィードバックを、自動情報提示の改善に活用

### 4. 症状別ナビゲーション（フォールバック機能）

**重要**: この機能も、作業セッション開始前や、自動情報提示でカバーできなかった場合の代替手段として位置づけます。

#### 4.1 症状カテゴリ
- エンジン（始動不良、アイドリング不安定、出力低下）
- ブレーキ（警告灯、ブレーキ鳴き、効き不良）
- バッテリー（バッテリー上がり、充電不良）
- 充電（充電開始不可、充電速度低下）
- 警告灯（各種警告灯の意味と対応）
- 異音（エンジン異音、足回り異音、ブレーキ異音）
- 電装（電装品不作動、通信不具合）
- ADAS（カメラ認識不良、レーダー不作動、警告表示）

#### 4.2 症状詳細ページ
- **安全確認**: 走行可否の目安、緊急対応の要否
- **警告灯の意味**: 表示内容の解説
- **まず確認すべき項目**: チェックリスト形式
- **確認手順**: ステップバイステップの診断フロー
- **関連マニュアル**: 該当する公式マニュアルのリンク
- **類似TIE事例**: 過去の解決事例
- **販売店内での対応履歴**: 自ディーラーでの過去の対応記録（プライバシー配慮）

### 5. 作業セッション画面（旧: 解決支援画面）

**重要**: この画面は、作業セッション開始後に常時表示される、整備士のメイン作業画面です。検索や症状別ナビゲーションではなく、この画面で作業を完結させることが目標です。

#### 5.2 自動提示情報セクション（工程に応じて動的に変化）

- **公式マニュアル**（該当ページを直接表示）
  - PDF埋め込み or HTML表示（該当ページのみ）
  - 作業手順の抜粋
  - トルク値、締付順序などの重要数値を強調表示
  - 警告事項を最上部に赤色背景で表示

- **類似TIE事例**（カード形式で複数表示）
  - 各事例のサマリー（車種、年式、症状、原因、対応、結果）
  - メーカー承認済みラベル
  - 「この事例が役立ちそう」ボタン（フィードバック収集）

  - チェックボックス形式（チェック状況を保存）

- **必要工具・部品リスト**（作業計画時に表示）
  - 工具名、部品番号、在庫確認リンク

#### 5.3 サイドバー: 関連情報（常時表示、スクロール可能）
- 過去の同一車両の作業履歴（自ディーラー内）
- この車種・年式でよくあるトラブル
- 最近更新された関連マニュアル・TIE
- 関連ニュース・リコール情報

#### 5.4 アクションバー（画面下部に固定表示）
- **工程を進める**ボタン（次の工程へ）
- **追加情報を探す**ボタン（AI検索画面へ遷移、フォールバック機能）
- **質問を投稿する**ボタン（問題交流へ遷移）
- **メーカーに問い合わせる**ボタン（問い合わせフォームへ遷移）

- **作業を完了**ボタン（作業完了フォームへ遷移）

## 非機能要件

### 1. パフォーマンス

#### 1.1 レスポンスタイム（初期1,000人規模を想定）
- **主要画面のレスポンスタイム**: 3秒以内（トップ画面、一覧画面、詳細画面）
- **検索レスポンスタイム**: 1秒以内（キーワード検索、絞り込み検索）
- **AIアシスト要約**: 5秒以内（OpenAI/Claude API呼び出し）
- **ページ遷移**: 1秒以内（Next.jsのルーティング最適化）

#### 1.2 同時アクセス
- **初期想定**: ピーク時100同時接続（1ユーザーあたり1日10分利用、業務時間帯集中と仮定）
- **1年後目標**: ピーク時1,000同時接続（100ディーラー、1,600ユーザー規模）

#### 1.3 スループット
- **検索クエリ**: 100クエリ/秒
- **ページビュー**: 1,000ページビュー/秒
- **API呼び出し**: 500リクエスト/秒

### 2. 可用性

#### 2.1 稼働率
- **目標**: 99.5%以上（月間ダウンタイム3.6時間以内）
- **業務時間帯（平日9:00-18:00 JST）**: 99.9%以上（月間ダウンタイム43分以内）

#### 2.2 メンテナンス
- **定期メンテナンス**: 月1回、深夜帯（2:00-4:00 JST）に実施、事前通知
- **緊急メンテナンス**: 障害発生時、即座に実施、事後通知

#### 2.3 障害復旧
- **RTO（目標復旧時間）**: 4時間以内
- **RPO（目標復旧時点）**: 24時間以内（1日分のデータ損失は許容）

### 3. セキュリティ

#### 3.1 認証・認可
- **パスワードポリシー**: 8文字以上、英数字記号混在、90日ごとに変更推奨
- **2要素認証**: Enterprise向けに提供（Google Authenticator、SMS）
- **セッション管理**: セッションタイムアウト30分（無操作時）

#### 3.2 データ保護

- **アクセス制御**: メーカー別データ分離（Row Level Security）、ディーラー別アクセス制御

#### 3.3 監査ログ

#### 3.4 脆弱性対策
- **依存関係の管理**: npm audit、Dependabotによる自動検出と更新
- **定期的なセキュリティスキャン**: OWASPZAPなどのツールによる脆弱性診断（四半期ごと）

## 技術スタック

### 推奨技術スタック

本プロジェクトでは、開発効率、スケーラビリティ、コスト、保守性を総合的に考慮し、以下のモダンな技術スタックを推奨します。

#### フロントエンド

- **フレームワーク**: Next.js 14（React 18ベース）
- **言語**: TypeScript 5
- **UIライブラリ**: Tailwind CSS + Headless UI
- **状態管理**: React Context API + Zustand（グローバル状態管理）
- **フォーム管理**: React Hook Form + Zod（バリデーション）

**選定理由:**
- Next.js: サーバーサイドレンダリング（SSR）、静的サイト生成（SSG）、ルーティング、API Routesを統合したフルスタックフレームワーク
- TypeScript: 型安全性によるバグ削減、IDEの補完機能、大規模開発での保守性向上
- Tailwind CSS: ユーティリティファーストのCSSフレームワーク、迅速なUI構築、一貫したデザインシステム
- 機能・非機能要件との整合性: レスポンシブデザイン、高速なページ遷移、SEO最適化に対応

#### バックエンド (BaaS)

- **BaaS**: Supabase（認証、データベース、リアルタイム機能、ストレージ）
- **認証**: Supabase Auth（メール認証、ソーシャルログイン、2FA）
- **ストレージ**: Supabase Storage（マニュアルPDF、画像、動画の保管）
- **リアルタイム**: Supabase Realtime（問題交流のリアルタイム更新）

**選定理由:**
- 認証、データベース、ストレージ、リアルタイム機能を一元管理し、開発効率を大幅に向上
- Row Level Security（RLS）によるメーカー別・ディーラー別のデータ分離を容易に実装
- サーバーレスアーキテクチャによるスケーラビリティとコスト最適化
- PostgreSQLベースのため、複雑なクエリ、全文検索、JSON対応が可能
- チームスキルへの適合性: 初期チームがバックエンド開発に不慣れでも、BaaSにより迅速に構築可能

#### データベース

- **データベース**: Neon（サーバーレスPostgreSQL）
- **全文検索**: PostgreSQL全文検索（tsvector、tsquery）
- **ベクトル検索**: pgvector拡張（意味的類似検索、AIアシスト機能用）

**選定理由:**
- サーバーレスPostgreSQLにより、スケーラビリティとコスト最適化を両立
- Supabaseとの統合が容易（SupabaseはNeonを推奨パートナーとして位置付け）
- ポイントインタイムリカバリー、読み取りレプリカ、自動スケーリングをサポート
- 全文検索とベクトル検索により、統合検索機能と意味的類似検索を実現
- スケーラビリティへの考慮: 初期規模（1,000人）から3年後目標（10,000人）までスムーズにスケール可能

#### ホスティング

- **ホスティング**: Vercel
- **CDN**: Vercel Edge Network
- **デプロイ**: GitHubと連携した自動デプロイ

**選定理由:**
- Next.jsとの最適な統合（Next.jsの開発元であるVercelが提供）
- 自動デプロイ、プレビューデプロイ、ロールバック機能による開発効率向上
- グローバルCDNによる高速な静的アセット配信
- エッジキャッシングによるレスポンスタイム短縮
- 可用性とパフォーマンス: 99.99%の稼働率、自動スケーリング、世界中のエッジロケーション

#### AI機能

- **AIモデル**: OpenAI API（GPT-4）または Claude API（Claude 3）
- **用途**: 
  - 要約（マニュアル・TIE・問題交流の横断要約）
  - 類似事例抽出（ベクトル検索との組み合わせ）
  - 検索補助（自然言語クエリの構造化）

**選定理由:**
- OpenAI GPT-4: 高精度な要約、日本語対応、APIの安定性
- Claude 3: 長文コンテキスト対応（100K tokens）、高い安全性、日本語対応
- 将来の拡張性: 翻訳、音声入力、チャットボット機能への拡張が容易

#### 検索基盤

- **全文検索**: PostgreSQL全文検索（tsvector、tsquery）
- **ベクトル検索**: pgvector拡張 + OpenAI Embeddings
- **検索アルゴリズム**: TF-IDF、BM25（PostgreSQLの標準機能）

**選定理由:**
- PostgreSQL全文検索: キーワード検索、フレーズ検索、ランキング機能をサポート
- pgvector: 意味的類似検索により、キーワードが合わない場合でも関連情報を検索可能
- 技術的制約への対応: キーワード検索だけでは症状・意図に対応しにくい課題を、ベクトル検索で解決

#### モニタリング・エラートラッキング

- **モニタリング**: Vercel Analytics
- **エラートラッキング**: Sentry
- **ログ収集**: Supabase Logs + Neon Logs

**選定理由:**
- Vercel Analytics: Web Vitals、ページビュー、ユニークビジターの可視化
- Sentry: エラー発生時のスタックトレース、ユーザー影響の把握、自動通知
- 保守性への考慮: モニタリングとエラートラッキングにより、障害の早期発見と迅速な対応が可能

- **DMS連携**: ディーラーマネジメントシステムとの連携
- **メーカー品質システム連携**: 品質データの双方向連携
- **AI要約の高度化**: GPT-4oによるマルチモーダル要約（画像・動画含む）
- **音声入力**: 整備現場での音声による作業記録、音声検索
- **多言語対応**: 英語、中国語、タイ語などへの対応（グローバル展開）

#### 成果物
- 診断機連携API
- 整備受付システム連携API
- マルチモーダルAI要約機能
- 音声入力機能
- 多言語対応（i18n）

**フェーズ分けの根拠:**
- Phase 4までに作業セッション管理とAI推薦エンジンが安定稼働している前提
- 外部システム連携により、作業コンテキストの入力を自動化し、整備士の負担をさらに削減
- 技術的依存関係: 診断機・整備受付システムとのAPI仕様調整が必要

## リスク分析

本プロジェクトにおける主要なリスクとその対策を、カテゴリー別に整理します。

### 技術的リスク

#### リスク1: 自動情報提示の精度不足
- **内容**: 作業工程に応じて自動提示される情報が的外れで、整備士が結局検索に頼ることになる
- **影響度**: 高（「探すをなくす」というコンセプトの根幹が崩れる、ユーザー満足度低下）
- **発生確率**: 高（Phase 2のルールベース実装では精度に限界がある）
- **対策**:
  - Phase 2でルールベースの最小構成を実装し、早期にフィードバック収集
  - 作業完了時のフィードバック収集を徹底（自動提示情報で十分だったか、追加で探した情報はあるか）
  - Phase 3でAI推薦エンジンを導入し、継続的に精度を改善
  - 自動提示精度のモニタリングダッシュボード（活用率、フォールバック利用率）
  - 「探す」をゼロにするのではなく、「80%削減」を現実的な目標として設定

#### リスク2: 作業コンテキスト入力の負担
- **内容**: 作業セッション開始時に車両情報・症状を入力する手間が増え、整備士が面倒に感じる
- **影響度**: 中（ユーザー体験の悪化、利用率低下）
- **発生確率**: 中
- **対策**:
  - 入力項目を最小限に絞る（車種、年式、VINのみ必須、症状は任意）
  - サジェスト機能・自動補完により入力の手間を削減
  - VIN入力により車両情報を自動取得（Phase 4）
  - 整備受付システム・診断機との連携により、入力そのものを自動化（Phase 5）
  - 作業セッション開始のメリットを明確に訴求（「30秒の入力で、10-20分の探索時間を削減」）
  
#### リスク3: AIの誤回答による整備事故
- **内容**: AI要約・回答が誤っており、整備士が誤った作業を行い、車両故障や人身事故が発生
- **影響度**: 高（PLリスク、ブランド毀損、法的責任）
- **発生確率**: 中
- **対策**:
  - AIは「最終回答者」ではなく「情報到達支援者」として位置付け
  - AI要約には必ず参照元マニュアル・TIE・問題交流のリンクを表示
  - 重要情報（警告・注意・禁止事項）は回答画面で強調表示
  - 免責表示の明記（AI要約は参考情報であり、最終判断は整備士の責任）
  - メーカー承認済み情報と非承認情報の明確な区別
  
#### リスク4: データ構造化の遅延
- **内容**: 既存マニュアル・TIE・問題交流のデータが構造化されておらず、タグ付けに時間がかかる
- **影響度**: 高（Phase 2の遅延、Phase 3以降の機能実装に影響）
- **発生確率**: 高
- **対策**:
  - Phase 1でデータ構造とタグ設計を優先的に検討
  - 既存データの構造を確認し、タグ付けの工数を見積もる
  - 自動タグ付け（AI）と人手タグ付けの併用
  - 優先度付け（サービスマニュアル、頻出TIE、解決済み問題交流から着手）
  
#### リスク5: スケーラビリティの不足
- **内容**: ユーザー数・データ量の増加により、レスポンスタイムが悪化
- **影響度**: 中（ユーザー満足度低下）
- **発生確率**: 中
- **対策**:
  - サーバーレスアーキテクチャ（Vercel、Supabase、Neon）による自動スケーリング
  - キャッシング戦略（Vercelエッジキャッシング、Redisキャッシング）
  - 読み取りレプリカ、データベースシャーディング（3年後目標）
  - パフォーマンスモニタリング（Web Vitals、Lighthouse）と継続改善

### ビジネス的リスク

#### リスク6: 収益化の失敗
- **内容**: ユーザー課金モデルやEnterprise分析機能が受け入れられず、収益が上がらない
- **影響度**: 高（事業継続の危機）
- **発生確率**: 中
- **対策**:
  - Phase 1でモックアップを共有し、顧客（メーカー、ディーラー）の反応を確認
  - 料金プランの柔軟な設計（Basic、Professional、Enterpriseの切り替え）
  - 既存顧客へのアップセル（Basic → Professional）とクロスセル（Enterprise分析機能）
  - 無料トライアル期間の提供（Professional、Enterprise）
  - 導入支援・カスタマーサクセスの強化
  
#### リスク7: 既存顧客の離反
- **内容**: 既存e-libraryから新システムへの移行により、ユーザー体験が変わり、顧客が離反
- **影響度**: 高（収益減少、評判悪化）
- **発生確率**: 中
- **対策**:
  - Phase 2でUI刷新時に、既存機能を維持しつつ段階的に移行
  - ユーザートレーニング（操作マニュアル、動画チュートリアル）
  - 移行期間中のサポート強化（問い合わせ対応、FAQ）
  - 既存顧客へのヒアリング（不満点、改善要望の収集）
  - 作業セッション機能は「オプション」として提供し、従来の検索も残す（移行期間中）
  
#### リスク8: 競合の出現
- **内容**: 類似サービス（OEM技術情報ポータル、修理情報データベース）が機能強化し、競争激化
- **影響度**: 中（市場シェア低下）
- **発生確率**: 中
- **対策**:
  - 差別化要因の強化（作業フロー駆動の自動情報提示という独自のUX、メーカー公式情報 + 現場知見の統合）
  - スイッチングコストの向上（顧客ごとのデータ資産蓄積、メーカー承認フロー組み込み）
  - ネットワーク効果の強化（ユーザー増加 → 知見増加 → 価値向上）
  - 継続的な機能拡張（AI、多言語、API連携）

### 法的/コンプライアンス的リスク

#### リスク9: PL（製造物責任）リスク
- **内容**: 誤った整備情報により車両故障や人身事故が発生し、法的責任を問われる
- **影響度**: 高（損害賠償、ブランド毀損）
- **発生確率**: 低（ただし、発生時の影響大）
- **対策**:
  - 免責表示の明記（AI要約・現場投稿・公式情報の区別）
  - メーカー承認ワークフローの厳格化（TIE、ニュース、マニュアル更新）
  - 情報の版管理（車種、年式、市場、マニュアル版）
  - 監査ログの記録（誰が、いつ、何を閲覧・投稿・承認したか）
  - 法務相談（利用規約、免責条項の整備）
  
#### リスク10: 個人情報漏洩
- **内容**: ユーザー情報、投稿者情報、ログデータが漏洩し、プライバシー侵害が発生
- **影響度**: 高（法的責任、信頼喪失）
- **発生確率**: 低（ただし、発生時の影響大）
- **対策**:
  - 通信暗号化（HTTPS）、データベース暗号化（Neon）
  - アクセス制御（Row Level Security）、メーカー別・ディーラー別データ分離
  - 監査ログの記録
  - セキュリティスキャン（OWASP ZAP）、ペネトレーションテスト（年1回）
  - プライバシーポリシーの整備

### 運用的リスク

#### リスク11: 障害対応の遅延
- **内容**: システム障害発生時に、迅速な復旧ができず、業務時間帯に長時間ダウン
- **影響度**: 中（ユーザー満足度低下、信頼喪失）
- **発生確率**: 中
- **対策**:
  - モニタリング・エラートラッキング（Vercel Analytics、Sentry）
  - アラート設定（エラー率、レスポンスタイム、可用性の閾値超過時に通知）
  - 運用マニュアル（障害対応手順、エスカレーションフロー）
  - 定期的な訓練（四半期ごとに障害復旧訓練）
  - バックアップ・リカバリー体制（ポイントインタイムリカバリー、過去30日分のバックアップ）
  
#### リスク12: サポート負荷の増大
- **内容**: 新システム導入後、問い合わせが増加し、サポート体制が追いつかない
- **影響度**: 中（ユーザー満足度低下、サポートコスト増加）
- **発生確率**: 高（新システム導入時は問い合わせ増加が一般的）

  - ユーザートレーニングの実施（オンライン/オフライン）

## UI/UXの方向性

### デザインコンセプト
**作業フロー中心、情報が届く、モダン、信頼感**

従来の「情報を探すシステム」から、**「情報が届くシステム」**へのパラダイムシフトを、UIで明確に表現します。作業セッション画面を中心に、整備士の作業フローに沿った情報提示を実現します。既存e-libraryは「古い閲覧システム」の印象があるため、UI刷新によりモダンで洗練された印象に変えます。一方で、自動車メーカー向けの公式情報閲覧基盤として、信頼感と安定感を損なわないよう配慮します。

### トーン＆マナー
**プロアクティブ、サポーティブ、効率的**

システムが「待っている」のではなく、整備士の作業を「サポートしている」印象を与えるトーンです。「次に○○を確認してください」「この情報が役立つかもしれません」といった、プロアクティブなメッセージングを多用します。問題交流（Q&A）など技術者同士のコミュニケーションの場では、親しみやすさや安心感も演出します。

### ターゲットデバイス
**PCメイン（作業セッション）、モバイル対応（クイック参照）**

整備現場ではPC利用が中心（デスクトップ、ノートPC、タブレット）であり、作業セッション画面はPCでの使いやすさを最優先します。一方で、外出先や移動中にマニュアルを確認するケースもあるため、モバイル（スマートフォン）でもクイック参照（過去の作業セッション閲覧、マニュアル閲覧）はできるようレスポンシブデザインを採用します。ただし、作業セッション開始やTIE作成など、複雑な操作はPC推奨とします。

### UI設計の基本方針

#### 1. 作業セッション画面を中心にする
- トップ画面では「作業を開始する」ボタンを最も目立つように配置
- 作業セッション画面が整備士のメイン作業画面となる
- 検索バーは「フォールバック機能」として控えめに配置

#### 2. 作業コンテキストを常に表示する
- ヘッダーに車種、VIN、症状、DTC、現在の作業工程を常時表示
- 作業経過時間を表示（時間管理の意識付け）
- 工程変更ボタンをヘッダーに配置（次の工程に進む、前の工程に戻る）

#### 3. 工程に応じた情報を自動的に変化させる
- 診断時 → 作業計画時 → 作業実施時 → 確認時で、表示される情報が動的に変わる
- 現在の工程で最も重要な情報を最上部に配置
- 「次に○○を確認してください」といったプロアクティブなメッセージ

#### 4. 情報種別をカードで分ける
- マニュアル、TIE、問題交流、ニュースをカード形式で表示
- カードに情報種別ラベル（色分け）、メーカー承認済みラベル、最終更新日を表示
- 「この情報が役立ちそう」フィードバックボタンをカードに配置

#### 5. 参照元と承認状態を必ず表示する
- AI要約には参照元マニュアル・TIE・問題交流のリンクを明示
- メーカー承認済み/未承認の明確なラベル表示
- 公式情報、ユーザー投稿情報の区別

#### 6. 警告・注意事項は常時表示で目立たせる
- 警告（赤色背景 + アイコン + 固定表示）
- 禁止事項（ストップマーク + 赤色テキスト）
- 作業前確認チェックリスト（チェックボックス付き、未チェック項目を強調）

#### 7. 次アクションを明確にする
- 画面下部のアクションバーに固定表示
- 主要アクション: 「工程を進める」「追加情報を探す」「質問を投稿する」「作業を完了」
- アクションの優先度を視覚的に表現（主要ボタンは大きく、色を目立たせる）

## まとめ

本システム要件定義書は、既存の情報閲覧サービス「e-library」を、**整備士が「探す」という行為をしなくても必要な情報が届く**技術情報活用プラットフォーム「e-library Next」へ進化させるための包括的な設計書です。

**主要な価値提案:**
1. **「探す」時間を80%削減**: 作業フロー駆動の自動情報提示により、検索する必要性そのものを削減（目標：1作業あたり10-20分 → 2-4分）
2. **作業フローの中断をゼロに**: 作業の各工程で必要な情報が自動的に表示され、作業に集中できる
3. **自己解決力を50%向上**: 作業タイミングに応じた情報提示により、問い合わせ前の自己解決を促進
4. **販売店課題を可視化**: 分析ダッシュボードにより、メーカーが販売店の困りごとをデータで把握
5. **収益性を向上**: 月額約2,000円から、ユーザー課金とEnterprise分析機能により、月額数十万円規模へ拡大

**推奨技術スタック:**
- Next.js + TypeScript + Supabase + Neon + Vercel
- 作業コンテキスト認識エンジン、AI推薦エンジン（OpenAI/Claude API）
- 開発効率、スケーラビリティ、コスト最適化、保守性を総合的に考慮

**段階的な開発ロードマップ:**
- Phase 1（1-2週間）: モックアップ・価値検証（「探すをなくす」コンセプトの提示）
- Phase 2（3-4ヶ月）: 作業コンテキスト管理・基盤構築（ルールベースの自動情報提示）
- Phase 3（2-3ヶ月）: AI推薦エンジン・精度向上（機械学習による推薦精度向上）
- Phase 4（2-3ヶ月）: メーカー向け分析・外部システム連携準備
- Phase 5（3-6ヶ月）: 診断機・整備受付システム連携（作業コンテキストの自動認識）

**リスク対策:**
- 技術的リスク（自動情報提示の精度不足、作業コンテキスト入力の負担、AIの誤回答、データ構造化）への対策を明確化
- ビジネス的リスク（収益化、既存顧客離反、競合）への対策を計画
- 法的/コンプライアンス的リスク（PL、個人情報漏洩）への対策を実施
- 運用的リスク（障害対応、サポート負荷）への対策を整備

**重要な方針転換:**
本要件定義書は、従来の「検索最適化」アプローチから、**「探すをなくす」**アプローチへの根本的な方針転換を反映しています。検索機能は、作業コンテキストベースの自動情報提示でカバーできなかった場合のフォールバック（補完機能）として位置づけます。

本要件定義書をベースに、Phase 1のモックアップ作成を進め、チーム・顧客からのフィードバックを踏まえて、要件を継続的にブラッシュアップしていきます。
