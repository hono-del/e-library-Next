## 1. 総合評価

### 評価レベル: **B（概ね良いが、いくつか明確化が必要）**

### 要件定義全体の完成度

詳細要件定義書は、システム構築に必要な要素を網羅的にカバーしており、全体として高品質である。特に、ビジネス要件、ユーザー要件、データベース設計、技術スタックの選定は優れている。

一方で、以下の観点で明確化が必要：
- **AI/レコメンドロジックの具体性が不足**（Phase 2のルールベースロジックが抽象的）
- **エラー・例外ケースの定義が不十分**
- **デモシナリオの具体性が弱い**
- **モックアップ実装の優先順位が不明確**

### 良い点

1. **「探すをなくす」という明確な価値提案**: 作業フロー駆動の情報提示は差別化要素として優れている
2. **ペルソナとユーザーストーリーの具体性**: 開発チームが理解しやすい
3. **データベース設計の詳細性**: ER図、テーブル定義、インデックス設計まで明記
4. **MVP定義と成功基準の明確性**: Phase 2完了時点の目標が具体的

### 主な懸念点

1. **Phase 2のルールベースロジックが抽象的**: 「DTCに対応するマニュアルページを抽出」とあるが、どのように対応付けるのか不明
2. **フィードバック収集後の改善フローが不明**: ログを取得した後、誰がどのように分析・改善するのか
3. **エラーケースが網羅されていない**: DTCが見つからない、複数のDTCがある、類似TIEが0件の場合など
4. **デモシナリオが抽象的**: ステークホルダーへの説明資料として使えるか不安

### Cursorでモックアップ実装に進める状態か

**条件付きで進めてよい**

以下を前提に、Phase 1のモックアップ実装に進むことを推奨：
- Phase 2のルールベースロジックをダミーロジックで代替（タグマッチングの簡易版）
- エラーケースは最小限（「該当する情報がありません」の表示のみ）
- デモシナリオを1つ具体化（佐藤健太のDTC診断シーンに絞る）

### ステークホルダー説明に使える状態か

**修正が必要**

以下を追加することで、説明資料として完成度が上がる：
- Before/Afterの具体的な作業時間比較（スクリーンショット付き）
- デモシナリオの具体化（セリフ・画面遷移・タイミング）
- Phase 2で実現する機能と、Phase 3以降の機能の明確な区分け

## 5. UXレビュー

### ユーザーの開始状況

**明確に定義されている**: 整備士が診断機でDTCを読み取った後、車両情報を入力して作業セッションを開始

### ユーザーの目的

**明確**: 車両の故障を診断し、修理計画を立て、作業を実施し、納車する

### ユーザーの行動フロー

**概ね良好、細部に改善余地あり**

**現状のフロー:**
1. トップ画面で「作業を開始する」ボタンをクリック
2. 車両情報入力フォームで、車種、年式、VIN、症状、DTCを入力
3. 作業セッション画面で、自動提示情報を確認
4. 工程を進めながら作業継続
5. 作業完了時にフィードバック入力

**改善提案:**

#### 改善1: 入力の負担軽減

**問題点**: 車両情報入力で、車種・年式・VIN・症状・DTCを手入力するのは負担が大きい

**改善案**:
- VIN入力時に、車種・年式を自動補完（VINの先頭8桁から判定）
- DTC入力時に、診断機から自動取得（Phase 5では診断機API連携、Phase 2ではコピー&ペースト）
- 症状入力は自由記述ではなく、プルダウン選択（「エンジン不調」「警告灯点灯」など）

#### 改善2: 工程変更のタイミング

**問題点**: 整備士が手動で「次の工程へ」ボタンを押すのは、押し忘れのリスクがある

**改善案**:
- 工程変更は手動でも可能だが、システムが「そろそろ次の工程に進みませんか？」と提案
- 提案タイミング: 診断フェーズで5分経過、マニュアルを3ページ閲覧、TIEを2件閲覧など

#### 改善3: フィードバック収集のタイミング

**問題点**: 作業完了時に「自動提示情報が役立ったか」を聞くのは、作業内容を忘れている可能性

**改善案**:
- 各工程完了時に、簡易フィードバック（👍/👎ボタン）を収集
- 作業完了時には、全体的な満足度のみを聞く

### 感情的な負荷

**概ね良好、注意点あり**

**良い点:**
- 自動提示情報により、「探す」負担が削減される
- 警告・注意事項の強調表示により、「見落とし」の不安が軽減される

**注意点:**
- **情報過多のリスク**: 自動提示情報が多すぎると、逆に「どこから見ればいいか分からない」という迷いが生じる
  - **対策**: 情報の優先順位を明確にし、「まずこれを確認」→「次にこれを確認」という順序を示す

- **誤った情報を信じるリスク**: AIアシスト要約が誤っている場合、整備士が誤判断する可能性
  - **対策**: 「この要約はAIによる参考情報です」という免責表示を常時表示

### 次の行動のわかりやすさ

**良好、さらに改善可能**

**現状**: 画面下部のアクションバーに「次の工程へ」「追加情報を探す」「質問を投稿」「作業を完了」ボタン

**改善案**:
- 現在の工程で推奨される次アクションを、ボタンの色・サイズで強調
- 例: 診断フェーズでは「次の工程へ（作業計画）」ボタンをオレンジ色で大きく表示

### ユーザーが迷いそうなポイント

#### 迷いポイント1: 作業セッションをいつ開始すべきか

**問題**: トップ画面に「作業を開始する」ボタンがあるが、整備士は「いつ押せばいいか」が分からない可能性

**対策**:
- ボタンの説明を追加: 「車両入庫後、診断機でDTCを読み取ったら開始してください」

#### 迷いポイント2: 自動提示情報と追加検索のどちらを先に使うべきか

**問題**: 自動提示情報を見る前に「追加情報を探す」ボタンを押してしまう可能性

**対策**:
- 作業セッション開始後、最初は「追加情報を探す」ボタンをグレーアウト
- 自動提示情報を10秒以上閲覧したら、ボタンを有効化

#### 迷いポイント3: 複数のDTCがある場合、どれから対処すべきか

**問題**: 3つ以上のDTCがある場合、優先順位が不明

**対策**:
- DTC重要度マスタを作成し、Highから優先的に表示
- または、システムが「まず P0420 から対処することを推奨します」と提案

### より自然な体験にするための改善案

#### 改善案1: 「探す」から「届く」への徹底

**現状の課題**: 作業セッション画面に、まだ「検索」の要素が残っている（画面下部の「追加情報を探す」ボタン）

**改善案**:
- 「追加情報を探す」ボタンを、「まだ分からないことがありますか？」という質問形式に変更
- ボタンを押すと、「何について知りたいですか？」という入力フォームが表示され、自然言語で質問できる

#### 改善案2: 作業の進捗を可視化

**現状の課題**: 作業セッション画面に、「あとどれくらいで作業完了か」が見えない

**改善案**:
- 画面上部に、作業工程の進捗バーを表示（入庫→診断→作業計画→実施→確認→納車）
- 現在の工程を強調表示し、次の工程を予告

#### 改善案3: 過去の作業履歴を活用

**現状の課題**: 作業セッション画面のサイドバーに「過去の作業履歴」があるが、活用方法が不明

**改善案**:
- 同じ車両の過去の作業履歴を、「前回の作業」として目立たせる
- 「前回は○○の修理をしています。今回の症状との関連性があるかもしれません」という提案を表示

## 6. AI・ロジックレビュー

### 6.1 Phase 2のルールベースロジック

#### 発火条件

**定義済み**: 作業工程変更時（診断→作業計画→作業実施）

#### 必要データ

**定義済み**:
- 車種、年式、VIN、症状、DTC（作業セッションテーブル）
- マニュアル（manuals テーブル）
- TIE（ties テーブル）
- 問題交流（qa_questions、qa_answers テーブル）

#### 判断ロジック

**抽象的、具体化が必要**

**現状の記述**:
```
IF 工程 = "診断" THEN
  - DTCに対応するマニュアルページを抽出
  - DTCに関連するTIE事例を抽出（関連度順）
  - DTCに関連する問題交流を抽出（解決済み優先）
  - DTCに関連する注意事項を抽出
```

**問題点**:
- 「対応する」「関連する」の定義が曖昧
- 「関連度順」「解決済み優先」の具体的なソート条件が不明
- 複数のDTCがある場合の処理が不明

**具体化した判断ロジック**:

```sql
-- 1. DTCに対応するマニュアルページを抽出
SELECT * FROM manuals
WHERE vehicle_model = '作業セッションの車種'
  AND model_year = '作業セッションの年式'
  AND '作業セッションのDTC' = ANY(related_dtc)  -- 配列マッチング
ORDER BY section ASC  -- セクション順（診断フロー → 作業手順）
LIMIT 5;

-- 2. DTCに関連するTIE事例を抽出
SELECT * FROM ties
WHERE vehicle_model = '作業セッションの車種'
  AND model_year = '作業セッションの年式'
  AND '作業セッションのDTC' && dtc  -- 配列の共通要素チェック
  AND approval_status = 'approved'  -- 承認済みのみ
ORDER BY 
  -- 関連度スコア計算（共通DTCの数 + 症状の類似度）
  (SELECT COUNT(*) FROM unnest(dtc) WHERE unnest = ANY('作業セッションのDTC')) DESC,
  created_at DESC  -- 新しい順
LIMIT 5;

-- 3. DTCに関連する問題交流を抽出
SELECT * FROM qa_questions
WHERE vehicle_model = '作業セッションの車種'
  AND '作業セッションのDTC' && dtc  -- 配列の共通要素チェック
  AND status = 'resolved'  -- 解決済み優先
ORDER BY 
  status DESC,  -- 'resolved' > 'open'
  created_at DESC
LIMIT 3;

-- 4. 複数DTCの処理
-- 複数のDTCがある場合は、各DTCごとに上記クエリを実行し、結果を統合
-- ただし、重複は除外し、関連度が高い順に表示
```

#### ユーザーへの説明

**不足、追加が必要**

**現状**: 自動提示情報が表示されるだけで、「なぜこの情報が表示されたのか」が不明

**改善案**:
- 各情報カードに、「表示理由」を追加
  - 例: 「このマニュアルは、DTC P0420 に対応しています」
  - 例: 「このTIE事例は、同じ車種・年式で、同じDTC P0420 が発生した事例です」

#### ユーザーの選択肢

**定義済み**: 自動提示情報を見る、追加情報を探す、質問を投稿

**改善案**:
- 各情報カードに、「役立った」「役立たなかった」ボタンを追加
- 「役立たなかった」を押した場合、「どの情報が不足していますか？」というフォローアップ質問を表示

#### 不適切な提案が出るリスク

**リスク**: 車種・年式が一致していても、市場が異なる場合（日本市場 vs 北米市場）、不適切な情報が表示される可能性

**対策**:
- vehiclesテーブルに`market`カラムがあるので、これを使った絞り込みを追加

## 9. 要件修正文案

### 修正案1: Phase 2のルールベースロジック

#### 元の要件

> **処理ロジック（Phase 2: ルールベース）**:
> ```
> IF 工程 = "診断" THEN
>   - DTCに対応するマニュアルページを抽出
>   - DTCに関連するTIE事例を抽出（関連度順）
>   - DTCに関連する問題交流を抽出（解決済み優先）
>   - DTCに関連する注意事項を抽出
> ```

#### 問題点

- 「対応する」「関連する」の定義が曖昧
- 「関連度順」の具体的な計算方法が不明
- 実装者が判断に迷う

#### 修正案

> **処理ロジック（Phase 2: ルールベース - タグマッチング）**:
>
> **前提条件**:
> - manuals、ties、qa_questions テーブルには、`related_dtc` (TEXT[]) カラムが存在
> - 作業セッションには、`dtc` (TEXT[]) カラムが存在
>
> **抽出ロジック（診断工程の場合）**:
>
> 1. **DTCに対応するマニュアルページを抽出**
>    - 条件: `vehicle_model`、`model_year` が一致、かつ `related_dtc` に作業セッションの `dtc` のいずれかが含まれる
>    - ソート: `section` 昇順（診断フロー → 作業手順の順）
>    - 件数: 最大5件
>    - SQL例: `WHERE vehicle_model = '車種' AND model_year = 年式 AND '作業セッションのDTC' = ANY(related_dtc) ORDER BY section ASC LIMIT 5`
>
> 2. **DTCに関連するTIE事例を抽出**
>    - 条件: `vehicle_model`、`model_year` が一致、かつ `dtc` に作業セッションの `dtc` のいずれかが含まれる、かつ `approval_status = 'approved'`
>    - ソート: 共通DTCの数（降順） → 作成日時（降順）
>    - 件数: 最大5件
>    - SQL例: `WHERE vehicle_model = '車種' AND model_year = 年式 AND dtc && '作業セッションのDTC配列' AND approval_status = 'approved' ORDER BY (共通DTC数) DESC, created_at DESC LIMIT 5`
>
> 3. **DTCに関連する問題交流を抽出**
>    - 条件: `vehicle_model` が一致、かつ `dtc` に作業セッションの `dtc` のいずれかが含まれる、かつ `status = 'resolved'`（解決済み優先）
>    - ソート: `status` 降順（resolved > open） → 作成日時（降順）
>    - 件数: 最大3件
>    - SQL例: `WHERE vehicle_model = '車種' AND dtc && '作業セッションのDTC配列' ORDER BY (CASE WHEN status = 'resolved' THEN 1 ELSE 0 END) DESC, created_at DESC LIMIT 3`
>
> 4. **注意事項を抽出**
>    - 条件: `warnings` テーブル（新規作成）から、`related_dtc` に作業セッションの `dtc` のいずれかが含まれる
>    - ソート: `severity` 降順（high > medium > low）
>    - 件数: 最大3件
>
> **複数DTCの処理**:
> - 作業セッションに複数のDTCがある場合、各DTCごとに上記クエリを実行
> - 結果を統合し、重複を除外
> - 共通DTCの数が多い順に表示

## 12. 最終判断

### 判断: **要件を修正してから開発に進むべき**

### 理由

詳細要件定義書は、全体として高品質であり、ビジネス要件、ユーザー要件、データベース設計は優れている。

一方で、以下の点で明確化が不足しており、このまま開発を開始すると、実装中に仕様確認が頻発し、開発期間が延びるリスクがある：

1. **Phase 2のルールベースロジックが抽象的**: 「DTCに対応するマニュアルページを抽出」とあるが、具体的なSQL文レベルのロジックが不明
2. **エラー・例外ケースが未定義**: DTCが見つからない、類似TIEが0件の場合など、エラーケースの挙動が未定義
3. **デモシナリオが抽象的**: ステークホルダーへの説明で使える具体的なシナリオが不足

### 推奨する修正内容

以下を要件定義書に追加・修正することで、Cursor開発に進める状態になる：

1. **Phase 2のルールベースロジックを具体化**（本レビューの「修正案1」を反映）
2. **AIアシスト要約の生成方法を明記**（本レビューの「修正案2」を反映）
3. **エラー・例外ケースの定義を追加**（本レビューの「修正案3」を反映）
4. **デモシナリオを具体化**（佐藤健太のDTC診断シーンを、セリフ付きのストーリーボード形式で作成）
5. **ダミーデータの準備**（車種マスタ、DTC定義マスタ、マニュアル、TIE、問題交流）

### 修正後の次ステップ

上記の修正を完了後、以下の順序で進めることを推奨：

1. **Phase 1モックアップの開発**（Cursorで実装、2週間）
2. **デモリハーサル**（社内メンバーへのデモ、フィードバック収集、1週間）
3. **ステークホルダーへのデモ**（顧客・役員へのプレゼン、Go/No-Go判断）
4. **Phase 2の開発着手**（基盤構築、3-4ヶ月）
